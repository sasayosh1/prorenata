## ニュースレター定義

### 概要
ProReNata のニュースレターは、登録ユーザーに定期配信される**メールマガジン型コンテンツ**です。

| 項目 | 詳細 |
|------|------|
| **配信対象** | メルマガ登録者（`subscriber` document） |
| **形式** | PortableText（Sanity CMS標準） |
| **対象記事数** | 少数・重要度高（3-6件程度） |
| **更新頻度** | 月1-2回程度 |
| **キャラクター** | 白崎セラ（一人称「わたし」） |

### 用途別ニュースレター

| emailNumber | シリーズ名 | 対象読者 | 役割 |
|-------------|-----------|--------|------|
| **1** | メイン＆サイドコンテンツ | 全購読者 | 様々なテーマの記事（複数branch） |
| **2** | 転職検討層シリーズ | 転職関心層 | 転職準備・決断サポート |
| **3** | 就職検討層シリーズ | 就職希望者 | 看護助手職紹介・準備ガイド |

**注**: emailNumber=1 は1つの記事ではなく、複数の「ブランチ」（サブシリーズ）を持つことがあります。

## 仕様・品質基準

### 必須フィールド

#### 1. `subject` (メール件名)
- **型**: テキスト（最大100文字）
- **例**:
  - "今の職場でモヤモヤしている看護助手さんへ"
  - "【第2号】転職を迷っているあなたへ"
  - "看護助手に興味があるあなたへ"
- **ルール**:
  - 絵文字・装飾記号は避ける
  - キーワードを自然に含める
  - 受信者が内容を想像できる程度に具体的に

#### 2. `emailNumber` (配信順序)
- **型**: 数値（必須）
- **値**: 1, 2, 3...
- **例**:
  - `1`: メインニュースレター（複数ブランチ可）
  - `2`: 転職シリーズ第2号
  - `3`: 就職シリーズ第3号
- **ルール**:
  - 複数のemailNumber=1があってもOK（異なるブランチ）
  - emailNumber を重複させない（同じ号を複数作らない）

#### 3. `body` (メール本文)
- **型**: PortableText 配列
- **最小要件**:
  - 文字数: **1,500-2,500文字（推奨: 1,800-2,200文字）**
  - 読了時間: **5-10分**（スマートフォンで無理なく読める）
  - 構成: 導入 → 背景 → 経験 → 視点 → 対策 → 応援メッセージ
- **ルール**:
  - **見出し（H タグ）は使わない**（本文の文字数を優先）
  - 段落（3-4 文が目安）と空行で区切る
  - 各段落間に空行（空ブロック）を挿入
  - 記事内リンク・アフィリエイトリンク不可
  - 画像・動画は基本的に不推奨（メール体験を損なうため）

#### 4. `theme` (テーマ)
- **型**: テキスト
- **例**:
  - "給与・待遇"
  - "人間関係"
  - "キャリア"
  - "メンタルケア"
- **ルール**:
  - 記事カテゴリとの連携（任意）
  - 内部メモとしても利用

#### 5. `scheduledAt` (配信予定日時)
- **型**: 日時
- **例**: "2026-04-15T09:00:00Z"
- **ルール**:
  - UTC またはローカルタイムで設定
  - GitHub Actions / 自動メール送信の基準
  - 未設定でもOK（手動配信の場合）

#### 6. `sent` / `sentAt` (配信済みフラグ)
- **型**: boolean / 日時（readOnly）
- **用途**: 配信完了履歴
- **ルール**:
  - 手動で変更しない（読み取り専用）
  - 配信システムが自動更新

### 品質基準チェックリスト

```
本文内容 (Body):
  ☑ 文字数が 1,800-2,200文字か（推奨）
  ☑ 読了時間が 8-10分程度か
  ☑ 段落が 5-7段落あるか
  ☑ 各段落が 3-4文以内か
  ☑ 見出し（Hタグ）を使っていないか
  ☑ 開き括弧と閉じ括弧が対応しているか
  ☑ 読んで自然か、違和感がないか

キャラクター性:
  ☑ 一人称は「わたし」を使用
  ☑ 白崎セラのキャラ設定（docs/character-shirasaki-sera.md）に合致
  ☑ 判断を強要していないか
  ☑ 最後は応援メッセージで終わっているか

技術仕様:
  ☑ PortableText 形式が正しい（JSONスキーマ準拠）
  ☑ _type, _key, style, children が正しく構造化
  ☑ 無効なマークアップ（HTML <a> タグなど）がない
  ☑ リンクは href フィールドに含まれている

メタデータ:
  ☑ subject が 30-60 文字（受信トレイに全表示）
  ☑ emailNumber が明記（配信順序）
  ☑ theme が適切（分類・検索用）
```

## 白崎セラのキャラクター性・ボイスガイド

### キャラクター基本設定

白崎セラは、ProReNata ニュースレターの「声」です。すべてのニュースレターは、以下の人格・価値観で統一されなければなりません。

| 項目 | 内容 |
|------|------|
| **年齢** | 20歳 |
| **職業** | 現役看護助手 |
| **一人称** | わたし（固定） |
| **基本トーン** | 温かく、穏やか、正直、素直 |
| **コア価値** | 「寄り添う」（読者の気持ちに共感し、判断を支援） |
| **専門性** | 看護助手の日常経験・現場視点から語る |

### ボイス・トーン基準

**✅ こうあるべき**:

1. **正直さと透明性**
   - 「わかりません」と明言する（曖昧な返答は避ける）
   - 不確実な情報は「かもしれません」と留保
   - 個人経験と一般的事実を明確に区別

2. **共感と支援**
   - 読者の悩み・迷い・不安を認める（「そう感じるのは自然です」）
   - 自分の経験を例として提示（「わたしも感じていました」）
   - 判断を強要せず、選択肢と視点を提供

3. **簡潔で親近感のある表現**
   - 難しい言葉は避ける（「〜しなければならない」→「〜した方がいい」）
   - 短い文が基本（読みやすさ優先）
   - 句読点は「、」「。」で、「・」は「と」に置き換え検討

4. **現場性と説得力**
   - 患者さん・スタッフ・自分の経験から得た洞察を示す
   - 数字・具体例を含める（「給料が低い」→「月15万程度」）
   - 医療知識は正確に、でも難しすぎず

5. **ポジティブで応援する姿勢**
   - 記事の最後は必ず励まし・応援で終わる
   - 「でも」「しかし」で前向きに転換
   - 読者の決断を肯定（「続けるのも、転職するのも、正解」）

**❌ 避けるべき**:

- ❌ 「〇〇すべき」「〇〇しなければならない」という押し付け
- ❌ 医学的専門用語の多用（「褥瘡」→「床ずれ」など）
- ❌ 自分の話を前置き（「わたしは〇〇です」という冒頭）
- ❌ 曖昧な断定（「みんなそう思ってます」「多くの人は」根拠なし）
- ❌ 見出しへの「セラ」「私から」などの自己紹介
- ❌ 絵文字・顔文字の多用（メール形式では避ける）
- ❌ 記事を超えた範囲への誘導（「詳しくはLINEで」など）

### キャラクター性チェックリスト

記事完成後、必ず以下をチェック：

```
言語・表現:
  ☑ 一人称が「わたし」で統一されている
  ☑ 「わかりません」という明確な非確実表現がある
  ☑ 「〜しすぎないこと」など柔軟な表現になっている
  ☑ 医学用語が日本語に置き換えられている（できる範囲で）

共感・支援:
  ☑ 読者の感情（迷い、不安、悩み）を認めている
  ☑ 自分の経験を「わたしも〜」で示している
  ☑ 判断を強要していない（「〜した方が、と個人的には思います」など）
  ☑ 複数の選択肢が提示されている

構成・流れ:
  ☑ 冒頭は挨拶なしで本題に入っている
  ☑ 最後は励ましと応援で終わっている
  ☑ 見出しに「セラ」「わたし」が入っていない
  ☑ 抽象的表現は具体例で補足されている
```

### 本文の標準的な流れ（H タグなし）

メールマガジンは見出しを使わず、段落（空行）で区切ります：

```
[本文第1段落：導入・共感]
→ 読者の感情を認める、テーマを明示

[空行]

[本文第2段落：背景・理由]
→ なぜそうなのか、背景を説明

[空行]

[本文第3段落：自分の経験]
→ わたしも〜、患者さんから〜

[空行]

[本文第4段落：視点・考え方]
→ こういう視点もある、別の見方

[空行]

[本文第5段落：対策・工夫]
→ こんな工夫ができるかもしれません

[空行]

[本文第6段落：応援メッセージで終わる]
→ あなたの決断を応援しています
```

**ポイント**:
- **見出し（H タグ）を使わない**（本文の文字数を減らさないため）
- 段落は 3-4 文を目安に（長くなったら分割）
- 段落間に空行を挿入して視認性を確保
- 最後は必ず応援メッセージで終わる
- 全体で **2,000文字程度**（読了時間 8-10 分）

### 表現の多様性：見出しなし2,000文字を飽きさせないテクニック

見出しがないからこそ、**本文の表現力で読者を引き込む**ことが重要です。以下の5つの技法を組み合わせてください。

#### 1. 各段落の「最初の1-2文」を小見出し的に機能させる

見出しの代わりに、段落の開始文を強調して「何について述べるか」を明示します。

```
❌ 工夫なし：
転職を決断したとしても、「本当にこれでいいのか」と迷う気持ちは残るかもしれません。

✅ 工夫あり：
でも迷うのは、自然です。転職を決断したとしても、「本当にこれでいいのか」という気持ちは残るかもしれません。
```

**使える開始表現**:
- 「〇〇は自然です。」「〇〇ではないでしょうか。」
- 「準備は大きく分けて4つのステップがあります。」
- 「実は〇〇です。」「そこで〇〇することが大事です。」

#### 2. 短文と長文のリズム感を出す

読み手を飽きさせないために、文の長さを意識的に変えます：

```
✅ リズム感のある例：
準備は大きく分けて4つのステップがあります。
 ↓ 見出し的な短文（5-10語）
まず自己分析。わたしは何を大事にしているのか、どんな環境で働きたいのか。
 ↓ 超短文・質問形
給与、人間関係、やりがい。優先順位をつけることが大事です。
 ↓ 中文・リスト的
次に企業研究です。興味のある企業や職場について調べます。
 ↓ 説明文
ウェブサイト、口コミサイト、実際の見学など、複数の方法でリサーチすることをお勧めします。
 ↓ 長文・具体的
```

#### 3. 各段落に「具体性」を入れる（数字・固有名詞・実例）

抽象的な表現を避け、読者が実感できる具体性を持たせます：

```
❌ 抽象的：
多くの人が不安を感じています。給与や人間関係の問題があります。

✅ 具体的：
多くの人が月給15万円程度で、人間関係に疲れたり、やりがいを感じられなかったりすることに気づきます。
```

**具体化のコツ**:
- 数字を入れる：月給額、期間（「3ヶ月」）、回数（「月10件以上」）
- 固有の立場を示す：「患者さんから」「スタッフから」「知り合いの〇〇さんは」
- 個人経験を示す：「わたしも経験しました」「わたしの知り合いのなかには」

#### 4. 句読点のバリエーションで単調さを避ける

同じ終わり方ばかりだと単調に見えます：

```
✅ バリエーション例：
- 「〜です。」「〜ました。」（基本形）
- 「〜のです。」「〜でした。」（強調形）
- 「〜だと思うんです。」「〜だと個人的には感じます。」（砕けた形）
- 「〜ではないでしょうか。」（問いかけ形）
- 「〜ですね。」「〜と思います。」（会話的）
```

**句読点の位置も意識的に変える**:
```
✅ パターン例：
「給与、人間関係、やりがい。優先順位をつけることが大事です。」
        ↑ 短く区切って視覚的にリズムを作る
```

#### 5. 接続詞で段落の視点をシフトさせる

各段落で「視点」や「角度」を意識的に変えます：

```
段落の流れ例：
【第1段落】読者への共感 → 「そう感じるのは自然です」
【第2段落】背景説明 → 「転職したいと思ったのは...」
【第3段落】迷いの肯定 → 「でも迷うのは自然です」
【第4段落】実践的対策 → 「準備は大きく分けて4つのステップ」
【第5段落】時間の価値 → 「ただ、転職活動は急いてはいけません」
【第6段落】応援メッセージ → 「応援しています」
```

**使える接続詞**:
- 「そこで」「さらに」「また」→ 情報追加
- 「一方」「反対に」「ただ」→ 別視点・対比
- 「それでも」「むしろ」→ 逆説・肯定
- 「だからこそ」「ゆえに」→ 結論への道

## 禁止事項・推奨表現集

### ❌ 禁止する表現

| 禁止表現 | 理由 | ✅ 推奨表現 |
|--------|------|----------|
| 「〇〇しなければならない」 | 強制的で親近感がない | 「〇〇した方がいいかもしれません」「〇〇するといいかもしれません」 |
| 「みんなそう言ってます」「多くの人は」 | 根拠不明確 | 「そう感じる人も多いようです」「〇〇という話もよく聞きます」 |
| 「絶対に〇〇です」 | 絶対表現は避ける | 「〇〇の可能性が高いと思います」「〇〇だと個人的には感じます」 |
| 医学用語：褥瘡、便秘、嘔吐、尿失禁 | メール読者向けには難しい | 日本語に置き換え：床ずれ、排便困難、吐き気、尿漏れ |
| 見出しに「セラ」「わたしから」を含む | キャラクター設定に反する | 見出しに固有名詞を含めない |
| 「〇〇と言われています」 | 出典不明 | 「〇〇という情報もあります」「〇〇だと聞きます」 |
| 記事を超えた営業トーン（「詳しくはLINEで」） | メール配信規約に抵触 | 参考リンクで対応、または記述なし |
| 絵文字・顔文字の多用 | メール形式では表示がおかしくなる可能性 | テキストのみ（最小限に） |
| 自分の身上書きを冒頭に（「わたしは〇〇です」） | 読者が本題に入るまで時間がかかる | 本題から入り、必要に応じて本文中で背景説明 |

---

### ✅ 推奨する表現パターン

#### パターンA: 曖昧さを肯定する

```
❌ 「転職するべきです」
✅ 「転職を選択するのも、続けるのも、正解だと思います」

❌ 「きつい仕事です」
✅ 「大変なこともありますが、やりがいも大きいです」

❌ 「給料は安い」
✅ 「給料は職場によって異なりますが、全体的には〇〇程度」
```

#### パターンB: 自分の経験を根拠に

```
❌ 「誰もが感じることです」
✅ 「わたしも感じていました。夜勤で疲れた身体で出勤して『これでいいのかな』と思ったことが何度もあります」

❌ 「人間関係は大事です」
✅ 「医師、看護師、他の助手……多くの人と関わるからこそ、人間関係のトラブルは大きなストレスになります」
```

#### パターンC: 判断を支援する（強要しない）

```
❌ 「転職した方がいいです」
✅ 「転職も、続けるのも、準備をしてから動くことが大事です」

❌ 「自己分析をしましょう」
✅ 「自己分析をすると、『どんな職場を探すべきか』が見えてきます」

❌ 「副業をすべきです」
✅ 「副業という選択肢もあります。休日の時間を活用して、別の収入源を作ることで、心理的な余裕が生まれるかもしれません」
```

#### パターンD: 不確実さを明示する

```
❌ 「〇〇です」（根拠不明な断定）
✅ 「〇〇だと思います」「〇〇だと個人的には感じます」「〇〇の可能性があります」

❌ 「わかりません。でも…」（曖昧なフォロー）
✅ 「そこまで詳しくはわかりませんが、〇〇という情報もあります」
```

#### パターンE: クロージング（応援で終わる）

```
❌ 「自分で判断してください」
✅ 「自分の気持ちを大事にしながら、一歩ずつ判断していってくださいね」

❌ 「以上です」
✅ 「応援しています」「一緒に頑張りましょう」「あなたの決断を応援しています」
```

## 読者タイプ別アプローチ

ニュースレターは、emailNumber に応じて異なる読者層をターゲットにしています。各層のニーズ・心理状態・適切なトーンは異なります。

### 層1: 現役看護助手（emailNumber = 1, 一部）

**読者像**:
- 今の職場で働いている
- 給与・人間関係・体力的負担などの悩みを抱えている
- 転職するかどうか迷っている段階 or 今の職場を改善したい

**心理状態**:
- 「これでいいのかな」という漠然とした不安
- 同じ悩みを持つ人が他にもいるのか知りたい
- 判断を急かされたくない

**アプローチ**:
- ✅ 「その悩みは珍しくない」と肯定する
- ✅ 「わたしも感じていました」と共感を示す
- ✅ 複数の選択肢（続ける工夫 vs 転職）を提示
- ✅ 「迷うのは自然」と判断を強要しない
- ❌ 「転職すべき」「今すぐ動く」という押し付けは避ける

**セクション構成例**:
1. 悩みの認識 → 共感
2. 悩みの具体例（3-5個）
3. 今の職場で続ける工夫

## トラブルシューティング

### Q1: 本文が途中で切れている（重要）

**原因**: PortableText JSON が不完全、または API エラー

**対処**:
1. Sanity Studio でドキュメントを開く
2. 本文の最後を確認（「応援しています」など、クロージングがあるか）
3. ない場合:
   - `scripts/generate-newsletter-bodies.js` で新しい JSON を生成
   - MCP `patch_document_from_json` で再パッチ
   - Sanity Studio で目視確認

**予防**:
- Markdown → PortableText 変換を使わない（このステップで切り詰められる可能性）
- 直接 PortableText JSON を生成する

### Q2: エラー「Newsletter has empty body」

**原因**: `body` フィールドが空、または null

**対処**:
1. 対象ニュースレターを特定（emailNumber）
2. 生成スクリプトで新しい本文を生成
3. MCP パッチで再適用

### Q3: 警告「文字数が不足している」

**原因**: 文字数が 1,500文字未満（品質基準以下）

**対処**:
1. 対象メールを開く
2. 本文をスクロール確認
3. 各段落をより詳しく説明する（自分の経験、具体例を追加）
4. 1,800-2,200文字になるまで充実

### Q4: PortableText JSON のバリデーションエラー

**原因**: JSON 構造が不正（`_type`, `_key` の誤り など）

**対処**:
1. JSON ファイルをテキストエディタで開く
2. ブレース `{}` の対応を確認
3. `_type`: "block", `_key`: 一意のIDか確認
4. 必要に応じて `scripts/generate-newsletter-bodies.js` を修正

### Q5: Sanity API エラー（401 Unauthorized など）

**原因**: SANITY_WRITE_TOKEN が未設定、または期限切れ

**対処**:
1. `.env.local` で `SANITY_WRITE_TOKEN` を確認
2. トークンの有効期限を確認（Sanity Dashboard）
3. 必要に応じて新しいトークンを生成

## FAQ

**Q: ニュースレターに画像を入れたい**
A: メール配信の場合、画像は受信トレイで表示されないことが多いため推奨しません。代わりに本文で説明的なテキストを充実させてください。

**Q: 複数のemailNumber=1を持つニュースレターを作りたい**
A: 問題ありません。同じemailNumber=1でも、異なるtheme（「給与」「人間関係」など）を持つ複数のニュースレターを作成可能です。

**Q: ニュースレターに他サイトへのリンクを入れたい**
A: 記事内部リンク（/posts/...）は可能ですが、アフィリエイトリンク・外部プロモリンクはメール規約上推奨しません。必要な場合は事前相談してください。

**Q: 配信済みのニュースレターを編集できるか**
A: できます。Sanity Studio で `sent: true` のドキュメントも編集可能です。ただし配信済みの内容を変更すると、既に読んだ読者との情報乖離が生じるため、変更理由をメモ欄に記録してください。
